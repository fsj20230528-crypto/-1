# 美食地图推荐店铺打卡设计

## 背景

现有页面的 `美食地图` 板块会基于定位、偏好和高德 POI 搜索结果生成推荐店铺列表，列表由 `#foodStops` 区域渲染。

当前页面中已经存在一个独立的 `打卡系统 / 通关文牒` 板块，但它记录的是“城市美食印章”的解锁状态，不是“推荐店铺”的打卡记录。

本次需求希望在 **不改其他代码** 的前提下，为美食地图中所有推荐店铺卡片增加一个 `打卡` 按钮，并支持本地持久化。

## 目标

- 在 `美食地图` 的每个推荐店铺卡片上增加 `打卡` 按钮
- 用户点击后，当前店铺进入“已打卡”状态
- 打卡状态持久化到本地，刷新页面或重新生成路线后仍可识别
- 改动收敛在推荐店铺卡片渲染及其最小配套逻辑内
- 不影响现有路线生成、地图展示和通关文牒功能

## 非目标

- 不将推荐店铺直接并入现有 `通关文牒` 页面的印章列表
- 不改动现有美食印章的本地存储结构和展示逻辑
- 不增加弹窗确认、额外抽屉、二级详情页
- 不接入后端服务，不做跨端同步
- 不改动美食路线的搜索、筛选、排序和地图绘制策略

## 现状与约束

### 现状

- 推荐店铺列表由 `renderFoodStops(start, stops, totalDistance, totalTime)` 统一渲染
- 每张推荐卡片当前包含：
  - 店铺序号与名称
  - 距离 / 评分 / 人均等元信息
  - 类型与地址提示
  - `在高德打开` 链接
- 现有打卡系统的数据来源为 `CHECKIN_CITIES` 和独立的 `zj_checkin_unlocked_v1`

### 约束

- 用户明确要求“不要改其他代码”
- 因此本次不改通关文牒版式，不复用城市美食印章的数据模型
- 推荐店铺来自高德 POI，与现有印章食物并非同一对象，不能强行共用 id

## 方案比较

### 方案 A：独立店铺打卡

为推荐店铺新增一套独立的本地打卡记录，只在店铺卡片中读写与展示状态。

优点：

- 改动最小，最符合“只改店铺卡片”
- 与现有通关文牒完全隔离，风险最低
- 能满足当前 `打卡` / `已打卡` 的核心诉求

缺点：

- 当前不会同步展示到通关文牒页

### 方案 B：并入现有通关文牒

点击推荐店铺打卡后，将店铺动态写入现有通关文牒页面。

优点：

- 业务上更像统一打卡系统

缺点：

- 必须改动现有打卡页结构、数据模型和渲染
- 与“不要改其他代码”的约束冲突

### 方案 C：仅保留当前页临时状态

点击按钮后仅改变当前页面状态，不做持久化。

优点：

- 实现最轻

缺点：

- 刷新或重新生成路线后状态丢失
- 不符合“加入打卡系统”的预期

## 结论

采用 **方案 A：独立店铺打卡**。

## 设计

### 1. 数据模型

新增一份独立的店铺打卡本地存储，例如：

- key：`zj_food_shop_checkins_v1`
- value：对象 map，按店铺稳定 id 存储

单条记录只保留当前功能所需的最小字段：

- `id`：店铺稳定 id
- `name`：店铺名
- `address`：店铺地址
- `lng`：经度
- `lat`：纬度
- `city`：可选，当前路线上下文中的城市信息
- `checkedAt`：打卡时间戳

### 2. 店铺稳定 id 生成

推荐店铺来自高德 POI，不能直接复用现有印章 id。

本次为推荐店铺独立生成稳定 id，推荐由以下字段拼接后标准化：

- 店铺名
- 地址
- 经纬度

目标是保证：

- 同一家店在刷新或重新生成路线后，仍能识别为同一条打卡记录
- 即使缺少个别非关键字段，也尽量生成可用 id

### 3. 渲染变更

继续由 `renderFoodStops()` 渲染推荐列表，只在单卡片内部新增：

- 一个按钮区域
- 一条可选的已打卡提示文案

按钮状态规则：

- 未打卡：显示 `打卡`
- 已打卡：显示 `已打卡`

已打卡卡片可补充一条轻提示，例如：

`已于 2026-06-18 09:32 打卡，记录已保存到本地。`

样式原则：

- 复用现有卡片风格
- 新按钮作为卡片内主操作，不与 `在高德打开` 链接挤在同一行
- 不引入新的复杂动效和额外交互层

### 4. 交互流程

用户点击 `打卡` 后：

1. 根据当前店铺生成稳定 id
2. 读取本地店铺打卡记录
3. 写入当前店铺记录与打卡时间
4. 立即更新当前卡片 UI 为 `已打卡`
5. 显示已保存提示

本次交互不包含：

- 页面跳转
- 二次确认弹窗
- 自动切换到通关文牒

### 5. 状态恢复

在推荐店铺卡片渲染时，先读取本地打卡记录，再决定当前店铺显示：

- `打卡`
- `已打卡`

因此只要同一家店再次出现在推荐结果中，就能自动恢复已打卡状态。

### 6. 容错处理

需要避免本地存储异常影响整个列表渲染。

建议处理方式：

- `load` 失败时回退为空对象
- `save` 失败时不阻断页面渲染
- 店铺字段不完整时，尽量基于已有关键字段生成 id
- 若确实无法生成稳定 id，则允许当前卡片降级为不可持久化状态，但不能导致推荐列表报错中断

### 7. 与现有打卡系统的关系

本次“推荐店铺打卡”属于新的独立子能力：

- 概念上属于打卡体系的一部分
- 技术上不接入现有 `CHECKIN_CITIES`
- 不写入 `zj_checkin_unlocked_v1`
- 不改 `renderCheckin()` 和通关文牒页面结构

这样可以保证边界清晰，也为后续若要把店铺打卡并入统一打卡页预留扩展空间。

## 测试与验证

本次以局部功能验证为主，重点检查：

- 每个推荐店铺卡片都能显示 `打卡` 按钮
- 点击后当前卡片立即变为 `已打卡`
- 刷新页面后重新生成路线，同一家店再次出现时仍显示 `已打卡`
- `在高德打开` 链接仍可正常使用
- 现有路线生成、地图展示、通关文牒功能不受影响

## 验收标准

- `美食地图` 中所有推荐店铺卡片都有 `打卡` 按钮
- `打卡` 点击后立即切换为 `已打卡`
- 打卡结果写入本地存储并可恢复
- 现有美食地图和通关文牒逻辑保持不变
- 改动范围控制在推荐店铺卡片及其最小配套逻辑内
