# 城市记忆来源分流设计

## 背景

当前“城市印象 -> 城市记忆”对所有县域都采用同一套策略：

- 有 DeepSeek API Key 时，调用 DeepSeek 生成 markdown 内容
- 生成结果缓存到 localStorage
- 无 Key 或生成失败时，回退到默认模板内容

现需求要求将杭州单独处理：杭州使用用户已提供的固定资料，其他城市继续沿用 DeepSeek 动态生成，并保持现有卡片版式不变。

## 目标

- 杭州的“城市记忆”固定使用内置资料
- 非杭州区县继续调用 DeepSeek 并按现有卡片版式展示
- 不改动现有卡片拆分、抽屉、封面图和章节图逻辑
- 保持现有 localStorage 缓存策略

## 方案

### 1. 新增固定资料来源

新增一份“城市记忆固定资料表”，先只包含杭州。

- key 使用地名，如 `杭州`
- value 为与现有渲染器兼容的 markdown 文本

该 markdown 仍然遵循当前版式要求：

- `#` 一级标题
- 多个 `##` 二级章节
- 允许 `###` 小节
- 支持段落与无序列表

### 2. 调整来源分流逻辑

在 `ensureCountyMemory(geoName)` 中增加“固定资料优先”的判断顺序：

1. 若命中固定资料表，直接写入 `persona.memory`
2. 标记为 `memoryEnriched = true`
3. 不调用 DeepSeek，不走 API，不写入 AI 缓存分支
4. 若未命中固定资料，再沿用现有缓存与 AI 生成逻辑

### 3. 保持展示层不变

以下逻辑保持原样：

- `renderMemoryCards()`
- 卡片摘要与详情抽屉
- 封面图与章节图生成方式
- 当前 tab/侧栏交互

也就是说，这次只切换“内容来源”，不改“展示版式”。

## 异常处理

- 杭州固定资料不依赖 API Key，也不受 DeepSeek 网络失败影响
- 非杭州区县：
  - 有缓存时优先读缓存
  - 有 Key 且无缓存时调用 DeepSeek
  - 失败时回退到默认模板 markdown

## 扩展性

后续如果用户继续提供宁波、绍兴、嘉兴等固定资料，只需要往固定资料表补充对应条目，不需要再重写生成链路。

## 验收标准

- 打开杭州时，城市记忆展示固定资料内容
- 打开非杭州区县时，继续按现有逻辑调用 DeepSeek 生成并展示
- 非杭州区县二次打开时，优先使用缓存
- 杭州与非杭州都保持现有卡片版式与详情抽屉交互
- 聊天功能与人格简介逻辑不受影响
